开篇词 (1讲)

开篇词｜掌握好学习路径，分布式系统原来如此简单

  • 分布式系统是互联网公司 IT 架构的事实标准

  • 站在不同的层面会看到不同的分布式系统

    • 单一应用层面：单体服务和多服务的分布式系统

    • 互联网公司层面：单机房、多机房或者跨洲际的分布式系统

    • 整个互联网层面：基于 TCP/IP 协议构建的整个互联网是一个超大规模的分布式系统

分布式系统的学习误区

  • 在做架构设计和选型的时候，没有知识系统做支撑，所以不能系统性地思考，处理问题也没有十足的把握

  • 不能从根本上直接解决由于分布式架构引入的问题

  • 初学者想要高效且系统性地掌握分布式系统，这本身就是一个悖论：

    • 理论与实践之间是有一道巨大的鸿沟
    • 论文、源码等资料的学习仅仅是入门，我们还需要结合业务场景、工作经历和实践经验，再加上思考与时间去慢慢沉淀才可以。

学习路径的重要性

  • 要有一条明确的学习路径，从最根本的原因出发
  • 对于初学者：要多问自己为什么，然后再思考怎么做
  • 在技术理解与实践中反复横跳，才能由点成线
  • 在总结中抓住技术实现的关键点和系统脉络，不迷失于细节，才能连线成网

课程设计

  • 找到分布式系统中稳定不变的知识、原理和解决思路
    • 注册发现原理
    • 故障处理的思路
    • CAP 理论
    • ......
  • 去繁从简
    • 不讨论分布式批处理和流计算
    • 只聚焦于日常工作中接触最频繁的在线业务分布式系统，根据是否有状态划分为：
      • 分布式计算
      • 分布式存储

专栏模块划分

概述篇

  • 学习一个知识应该先理解这个知识的来龙去脉。先讨论分布式系统产生的过程：它为什么会产生，产生后解决了什么问题，又带来了哪些新问题，遇到哪些方面的挑战。
  • 抓住分布式系统的脉络和关键点，可以有很强的学习目标和路径，就不会迷失在各种系统和框架实现的细节中。

分布式计算篇

  • 从单机系统演进到分布式系统后，引入哪些新问题的角度，在技术原理层面一个一个讨论并解决这些问题。
  • 收获：可以在各种系统和场景中理解和运用这些知识，并且知道在系统设计层面应该如何取舍。

分布式存储篇

  • 从简单到复杂，一起讨论数据分片、数据复制、分布式事务和一致性等相关的知识
  • 收获：在做架构设计时，思维的深度和广度都得到了提升

总结篇

  • 了解分布式系统的发展历程和未来趋势
  • 分布式计算的角度：一起讨论分布式系统是怎么从单机系统演进到 Service Mesh
  • 分布式存储的角度：一起讨论分布式系统是怎么从单机系统的 ACID 演进到 NewSQL



概述篇 (3讲)

01｜导读：以前因后果为脉络，串起网状知识体系

学习方法

  • 学习一个知识之前，先理解它的前因后果，才能比较好地抓到它的脉络和关键点，不至于迷失在细节中
    • 前因：这个知识产生的过程中，解决了什么问题？怎么解决的？
    • 后果：它的出现带来了哪些新问题？

前因：分布式系统解决了什么问题？怎么解决的？

分布式系统解决的问题

  • 在互联网时代，单机系统是无法解决成本、效率和高可用问题的，分布式系统应运而生
    • 分布式系统解决了单机性能瓶颈导致的成本问题
    • 解决了用户量和数据量爆炸性地增大导致的成本问题
    • 想要提供高可用的服务，单机系统也会变成一个分布式系统
    • 分布式系统解决了大规模软件系统的迭代效率和成本的问题

分布式系统是如何解决问题的

  • 将一些廉价的 PC 机，通过网络连接起来共同完成工作，并且在系统中提供冗余，来解决高可用的问题。（通过多工作节点，来解决单机系统面临的成本、效率和高可用问题的。）
  • 分布式系统的定义：分布式系统指的是由一组通过网络进行通信，为了完成共同的任务，而协调工作的计算机节点组成的系统

理解分布式系统

  • 我们可以把分布式系统看成是单体系统的平民化和物美价廉的版本
  • 分布式系统在不同的业务场景中有着不同的表现形式
    • 流量路由策略加多副本部署（微服务是其中的一种架构形式）是无状态服务的分布式架构方案
    • Redis Cluster 和 Codis 等方案实现了缓存的分布式化
    • Kubernetes 实现了操作系统的分布化
    • NewSQL 数据库的分布化

后果：如何思考和处理分布式系统引入的新问题

  • 引入了分布式系统内部工作节点的协调问题，主要体现在分布式系统内部组件、实例之间，通过异步网络进行通信和协调的问题上

针对内部工作节点的协调问题，分布式系统是怎么做的

对于分布式计算（无状态）的情况围绕 7 个子问题进行思考

  • （1）怎么找到服务？
    • 服务注册与发现机制
    • 理解服务注册发现机制的实现原理
    • 思考：服务注册发现，是选择做成 AP 系统还是 CP 系统更合理
  • （2）怎么找到实例？
    • 服务实例是完全对等的（无状态）：按负载均衡的策略来处理（轮询、权重、Hash、一致性 Hash，FAIR 等各种策略的适用场景）
    • 服务实例不对等（有状态）：通过路由服务（元数据服务等）先确定要访问的请求数据做到哪一个实例上，然后再进行访问
  • （3）怎么管理配置？
    • 通过配置文件管理：低效、易出错
    • 一般的思路是通过一个中心化的存储来统一管理系统的配置，即配置中心
  • （4）怎么进行协同？
    • 功能模块都在一个进程中：调用系统的 API 加锁
    • 分布式系统中：通过一个跨进程与机器的分布式锁来解决
  • （5）怎么确保请求只执行一次？
    • 重试加上幂等
  • （6）怎么避免雪崩？
    • 快速失败和降级机制（熔断、降级、限流等）：快速失败会导致部分的请求失败，快速失败会带来系统数据不一致的问题
    • 弹性扩容机制：弹性扩容的实现成本比快速失败要大，响应时间也更长
  • （7）怎么监控告警和故障恢复？
    • 一定要完善分布式系统的监控（比如接口的时延和可用性等信息），分布式追踪 Trace ，模拟故障的混沌工程以及相关的告警等机制。同时做好故障恢复预案

分布式存储（有状态）的情况，围绕 4 个子问题思考

  • （1）在 CAP 及其相关理论与权衡方面，需要了解 ACID 、 BASE 和 CAP 理论，参考文献如下

    • 英文版本
    • 中文版本

  • （2）怎么做数据分片？

    • Hash 和 Region 分片的策略（要注意了解它们的优缺点和各自的应用场景）

  • （3）怎么做数据复制？

    • 中心化方案（主从复制、一致性协议，比如 Raft 和 Paxos等）
    • 去中心化的方案（ Quorum 和 Vector Clock）
    • 要了解它们的优缺点，各自的应用场景，以及对系统外部表现出来的数据一致性级别（线性一致性、顺序一致性、最终一致性等）

  • （4）怎么做分布式事务？

    • 对于分布式系统来说，要实现事务，首先需要对并发事务进行排序的能力（这样在事务冲突的时候，就可以确认哪个事务提交成功，哪个事务提交失败）
    • 单机系统中：过时间戳加序号的方式就可以实现
    • 分布式系统：参考Google 的 Spanner ，它是通过 GPS 和原子钟，实现 TrueTime API 来解决生产事务ID，从而实现全球分布式数据库的。有了事务 ID 后，通过 2PC 或者 3PC 协议来实现分布式事务的原子性

系统学习方法

  • 从实践出发：
    • 究目前比较常用的分布式系统的设计：HDFS、GFS、Kafka、Pulsar、Redis Cluster 和 Codis、MySQL 的分库分表（传统关系型数据库的分布式方案）、MongoDB 的 Replica Set 和 Sharing 机制集（NoSQL 数据库）、TiDB（NewSQL）以及一些微服务框架等
  • 从理论出发：
    • 研究分布式相关的论文
    • 《数据密集型应用系统设计》
  • 如何处理分布式系统引入的，内部工作节点的协调问题：
    • 路由问题
      • 怎么找到服务、怎么找到实例、负载均衡（正常情况下的路由问题）
      • 怎么做数据分片（带状态的路由问题）
      • 怎么避免雪崩（异常情况下的路由问题）
    • 共识问题
      • 怎么做数据复制
      • 怎么做分布式事务
      • 怎么做分布式锁
    • 运维问题
      • 怎么管理配置
      • 怎么监控告警和故障恢复

小结

  • 点、线结合，形成网状的知识体系


02｜新的挑战：分布式系统是银弹吗？我看未必！



03｜CAP 理论：分布式场景下我们真的只能三选二吗？

什么是 CAP 理论

  • CAP 理论是加州理工大学伯克利分校的 Eric Brewer 教授在 2000 年 7 月的 ACM PODC 会议上首次提出的，它是 Eric Brewer 在 Inktomi 期间研发搜索引擎、分布式 Web 缓存时得出的关于数据一致性（ C：Consistency ）、服务可用性（ A：Availability ）、分区容错性（ P：Partition-tolerance ）的一个著名猜想：

It is impossible for a web service to provide the three following guarantees :

Consistency, Availability and Partition-tolerance.

一致性（ C ）

  • CAP 理论中的一致性是指强一致性（ Strong Consistency ），又叫线性一致性（Linearizable Consistency ）它要求多节点组成的分布式系统，能像单节点一样运作
  • 一致性中除了强一致性之外，还有其他的一致性级别：
    • 序列一致性（ Sequential Consistency ）
    • 最终一致性（ Eventual Consistency ）

可用性（ A ）

  • 要求系统提供的服务必须处于 100% 可用的状态，对于用户的每一个操作请求，系统总能够在有限的时间内返回结果
    • 100% 可用：系统必须完全可用，不允许任何不可用的情况出现
    • 有限时间内：对于客户端的一个请求，系统必须在指定的时间内返回对应的请求结果
    • 返回结果：系统在完成对客户端请求的处理后，必须返回一个正常的响应结果

分区容错性（ P ）

  • 在整个分布式系统中，因为各种网络原因，系统被分隔成多个单独的部分，它不仅包含我们通常说的网络分区，也包含因为网络丢包导致的网络不通的情况。
  • 在分布式系统实现中，分区容错性 P 是一个必须项

CAP 理论的影响与思考

  • 基于 BASE 理论支撑的 NoSQL 运动坚持创造各种可用性优先、数据一致性其次的方案
  • 传统数据库则坚守 ACID 特性（原子性、一致性、隔离性、持久性），优先数据一致性
  • etcd 属于 CP 模型
  • 可以将 CAP 理论的选择，推迟到出现网络分区的时候而不是系统一启动就进行 CAP 的选择
  • CAP 理论给我们定义了系统的设计边界，虽然想要设计出超过边界的系统是徒劳的，但是我们却可以无限逼近边界，并且把它作为我们设计系统的目标
  • 对于 CAP 理论，我们不会简单地三选二或者二选一。对于 AP 模型的系统，我们会努力去提升数据一致性的级别，而对于 CP模型的系统，我们会努力去提升系统可用性的级别

分布式计算篇 (15讲)

04｜注册发现： AP 系统和 CP 系统哪个更合适？

分布式计算技术体系

  • 分布式批处理技术：MapReduce
  • 分布式流计算技术：Flink
  • 分布式在线业务技术：Istio

为什么需要服务注册发现

  • 所以我们决定按资源和业务等维度对单体服务进行拆分

服务注册发现的业务场景

  • 服务 A 获取被调用服务 B 的 IP 和 Port 就是服务注册发现的业务场景

服务注册发现的关键问题

  • 可以将配置 IP 和 Port 列表的方式修改为配置域名和 Port
  • 服务注册发现需要解决的两个关键问题：
    • 统一的中介存储：调用方在唯一的地方获得被调用服务的所有实例的信息
    • 状态更新与通知：服务实例的信息能够及时更新并且通知到服务调用方

如何选择适合的中介存储

  • 可用性要求非常高：因为服务注册发现是整个分布式系统的基石，如果它出现问题，整个分布式系统将不可用
  • 性能要求中等：只要设计得当，整体的性能要求还是可控的，不过需要注意的是性能要求会随分布式系统的实例数量变多而提高
  • 数据容量要求低：因为主要是存储实例的 IP 和 Port 等元数据，单个实例存储的数据量非常小
  • API 友好程度：是否能很好支持服务注册发现场景的“发布 / 订阅”模式，将被调用服务实例的 IP 和 Port 信息同步给调用方
  • 希望在系统出现网络分区的时候，调用方一定不能获取过期的被调用服务实例信息，那么就选择 etcd 和ZooKeeper
  • 认为获取过期的实例信息，可能比完全不能获取被调用服务的实例信息要好，那么就选择 Eureka

怎么做服务状态的更新与通知

  • 服务注册（服务的状态更新）：

服务的每一个实例每隔一段时间，比如 30 秒，主动向中介存储上报一次自己的 IP 和 Port 信息，同时告诉中介存储这一信息的有效期，比如 90 秒。这样如果实例一直存活，那么每隔 30秒，它都会将自己的状态信息更新到中介存储。如果实例崩溃或者被 Kill 了，那么 90秒后，中介存储就会自动将该实例的信息清除，避免了实例信息的不一致。所以这里的数据同步是最终一致性的

  • 服务发现（服务的状态通知）：

服务的调用方通过中介存储监听被调用服务的状态变更信息。这里可以采用“发布 /订阅”模式，也可以采用轮询模式，比如每 30 秒去中介存储获取一次。所以这里的数据同步也是最终一致性的

选择 AP 还是 CP

  • 从服务注册发现的场景来说，我认为 Eureka 之类的 AP 系统更符合要求（因为服务发现是整个分布式系统的基石，所以可用性是最关键的设计目标）
  • 在服务发现的场景里面，高可用性是最应该去考虑的设计指标，所以选择 AP系统做中介存储是一个不错的选择
